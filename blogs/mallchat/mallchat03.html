    vuePress-theme-reco ShowedMaker    2024

    单聊群聊技术方案

    ShowedMaker 9/2/2023 数据库，项目结构，前后端交互

    # 联系人

    # 批量校验查看是否是自己好友

    点击用户头像加好友，需要先判断是不是自己的好友。用该接口

    # 申请好友

    点击头像发起申请，填写申请信息。会有一个ws推送给对方

    # 删除好友

    调用接口删除好友

    # 申请列表

    根据创建时间，普通分页。只能接受，不能拒绝

    只给uid，信息自己获取。展示申请信息。

    # 记录未读数

    # 计算好友会话接口

    根据uid从后端拿会话

    # 同意好友申请（不需要拒绝，冷处理）

    同时给该用户，添加对于好友的会话。

    后台会推送一条新消息（同意了好友申请）

    # 联系人列表（支持游标翻页）

    可简单展示，直接翻页

    发消息可定位发送（暂时不做）

    会有新消息触发，推送给两个人。如果这时候没有roomId，要去后端加载会话信息。

    # 会话列表

    # 会话列表（有未读数，根据时间线游标翻页）

    /capi/chat/public/contact/page

    返回的是房间id，单聊群聊都有一个房间id。

    除了第一次加载。后续每次有新消息，都要更新会话列表的activeTime，如果该会话不在当前房间内，还需要更新未读数。unreadCount+1.`

    # 会话详情

    有些消息推送给用户，用户会话列表没有该消息的会话。可以查详情 /capi/chat/public/contact/detail

    会话未读点进去。如果未读超过一页，加载不完，需要展示新消息跳转图标，去后端加载（优先级低）

    # 消息已读未读列表

    查询消息的已读或者未读用户列表，游标翻页 /capi/chat/msg/read/page

    # 消息已读未读数

    自己发送的消息，可以查看已读未读数和已读未读列表。

    自己发送消息后，前端加个定时任务，5分钟内每10s中查询一次已读数。

    用户点开列表的时候，再加载未读数。

    获取消息列表，消息列表是自己的消息，来后端批量获取已读数。

    如果点进去查看

    1.去后端获取消息未读数

    2.获取一个已读或未读的列表。

    新消息发送完毕后。

    当前会话会启动一个定时器。每30s查出当前屏幕内的自己的消息。调用后端获取消息已读数，进行更新。超过两天的消息定时器就不跑了。

    # 已读上报

    用户在某个房间获取了新消息，需要上报最新阅读状态。如果用户新打开某个房间看了新消息，也需要上报阅读状态。每次有消息推送，用户查看了，也要上报阅读状态。

    /capi/chat/msg/read

    上报时间点：

    1.用户首次进入会话，加载完了消息，触发上报

    2.在当前会话内的每条新消息，接收到之后，进行上报。

    # 群组管理

    # 好友详情

    根据详情判断能否发消息。

    # 群详情信息

    返回群的基本信息，在线人数，用户角色（关系到踢人和撤回）返回4代表用户已被移除群

    废除/capi/chatpublic/member/statistic 新接口：/capi/room/public/group

    # 创建群聊

    选择了人自动成群。

    创建成功后，返回roomId。同时后端会有一条系统消息，推送所有群成员。谁邀请了谁加入群聊。

    系统消息有roomId。

    # 群成员列表

    /capi/room/public/group/member/page

    # 艾特人员列表

    /capi/room/group/member/list

    # 踢出群成员

    /capi/room/group/member

    踢出群成员。权限校验，要么是群级别的管理员等，要么是全局的管理角色。

    用户角色来自于群详情信息

    # 自己退群

    暂不开放

    # 邀请新成员加入

    /capi/room/group/member post

    前端：根据群成员的艾特列表拿到的uid，和邀请界面的好友列表做去重判断。

    校验自己是这个群的用户，校验邀请的都是自己好友。

    新加入的成员，后端会给ws推送

    # ws推送

    # 申请推送

    只需要给前端推送当前的未读数就够了，用户自己点进去的时候看待处理的。

    com.abin.mallchat.custom.user.domain.vo.response.ws.WSFriendApply

    type=10

    # 群成员变动

    新com.abin.mallchat.custom.user.domain.vo.response.ws.WSMemberChange type=11
